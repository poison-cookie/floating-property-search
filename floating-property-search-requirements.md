# 管理会社サイト共通検索ランチャー 要件定義・仕様書

## 1. プロジェクト概要

本プロジェクトは、管理会社サイトごとに異なる検索ページ、検索欄、検索ボタン、検索URLの違いを吸収し、対象サイト内のどのページからでも物件名などの検索欄へ素早く移動・フォーカスできる Tampermonkey / Violentmonkey 用ユーザースクリプトを作成するものである。

既存プロジェクト「Site Input Autocomplete」は入力欄への候補表示・入力補助を目的とする。一方、本プロジェクトは「検索欄へ戻る操作を共通化する検索フォーカスランチャー」として設計する。

初期版では検索実行の完全自動化よりも、`Ctrl + /` などのショートカットで対象サイトの検索欄へフォーカスすることを中核機能とする。検索ページ以外にいる場合は、事前に選択した検索対象タブへ切り替える、または検索ページを新しいタブで開いて検索欄へフォーカスする。

## 2. 既存プロジェクトとの関係

### 2.1 既存プロジェクトの役割

既存の入力補助ツールは、以下を主目的とする。

- サイトごとの入力候補表示
- 使用回数、最終使用日、ピン止めによる候補管理
- CSV / JSON による候補移行
- Chrome標準候補の抑制
- 入力欄への安全な値反映
- React / Vue 系サイトへの入力反映補助

### 2.2 本プロジェクトの役割

本プロジェクトは、以下を主目的とする。

- 対象サイト内の任意ページから検索欄へ移動・フォーカスする
- サイトごとの検索方式を吸収する
- 検索対象タブ、検索ページURL、検索欄セレクタの違いを吸収する
- 検索欄まで戻る手間、スクロールする手間を減らす
- 管理会社サイトごとの検索仕様を設定として管理する

### 2.3 分離する理由

両者は近い領域のツールだが、責務が異なるため別プロジェクトとして管理する。

- 入力補助ツールは「入力欄での候補表示」が主目的
- 本ツールは「検索欄へのフォーカスと画面移動」が主目的
- 検索ランチャーはページ遷移や、後続機能ではボタンクリックを伴う可能性があるため、安全設計を独立して行う必要がある
- サイト別検索設定は保守対象が増えやすく、既存ツールに混ぜるとコードの複雑度が上がる
- UI常駐領域が増えるため、既存ツールと競合しない表示設計が必要

### 2.4 フォーカス優先設計の利点

初期版では、検索欄へのフォーカスを主目的にする。

この方針により、イタンジのようなReact / Vue / 独自JSで入力欄や検索ボタンを強く制御しているサイトでも、検索実行の自動化に比べて壊れにくくなる。検索欄にフォーカスした後の入力や検索実行はユーザー操作に任せるため、意図しないボタンクリック、値反映失敗、サイト側イベント未発火のリスクを抑えられる。

後続機能として自動入力・自動検索を追加する場合も、サイトごとの明示設定がある場合だけ有効にする。

## 3. 基本方針

### 3.1 実装形式

- Tampermonkey / Violentmonkey 用ユーザースクリプトとして実装する
- JavaScriptのみで実装する
- 初期版では外部ライブラリを使用しない
- 全サイト共通スクリプトとして作成する
- 実際に有効化するサイトはサイト別設定で制御する
- 未設定サイトではUIを表示しない

### 3.2 既存プロジェクトと競合しない設計

- スクリプト名、保存キー、DOM ID、CSSクラス名は既存プロジェクトと完全に分離する
- 既存プロジェクトの保存キー `mansionAutocomplete.*` は使用しない
- 本プロジェクトの保存キーは `floatingPropertySearch.*` で統一する
- UIは専用の Shadow DOM 内に作成する
- 既存ツールの右下 `設 定` ボタンと重ならないよう、初期表示位置は左下とする
- z-index は必要最小限にし、候補ポップアップとの重なりを避ける
- 本ツールのCSSがサイト側や既存ツールへ影響しないようにする

### 3.3 将来連携できる設計

初期版では既存ツールと直接依存しない。ただし、将来以下の連携ができるように設計する。

- 検索ランチャーの入力欄に、既存ツールの候補データを表示する
- 既存ツールの検索正規化ロジックを共有する
- 既存ツールで登録したサイト別候補を、検索ランチャーでも利用する
- 検索ランチャーで検索実行した語句を、既存ツールの使用回数へ反映する
- JSON移行時に両ツールの設定をまとめてバックアップできるようにする

連携は後方互換性を崩さないよう、明示的な設定で有効化する。

## 4. 目的

本ツールの目的は以下である。

- 管理会社サイトごとの検索UI差異を吸収する
- 検索ページへ戻る手間を減らす
- 詳細ページ、一覧ページ、別画面からでもすぐ物件名検索できるようにする
- スクロールして検索欄を探す手間を減らす
- サイトごとの検索仕様を設定として蓄積する
- 社内で共通の検索設定を共有できるようにする

## 5. 対象外

初期版では以下を対象外とする。

- ログイン処理
- 複数条件検索
- 自動連続検索
- 検索結果の自動取得
- 検索結果の保存
- 図面OCR
- 物件情報の自動登録
- 大量アクセスを伴う巡回
- サイト規約を回避する処理
- CAPTCHAや二段階認証の突破

本ツールは、あくまで利用者がブラウザ上で行う検索操作を補助する目的に限定する。

## 6. UI仕様

### 6.1 表示位置

初期表示位置は画面左下とする。

```css
position: fixed;
left: 14px;
bottom: 14px;
```

既存入力補助ツールの右下設定ボタンと競合しないため、左下を標準位置とする。

### 6.2 表示条件

- 対象サイト設定が存在する場合のみ表示する
- 未設定サイトでは何も表示しない
- サイト別に無効化されている場合は表示しない
- Tampermonkeyメニューから再有効化できるようにする

### 6.3 パネル内容

検索パネルには以下を表示する。

- ツール名
- 現在の対象サイト名
- キーワード入力欄
- 検索ボタン
- 最小化ボタン
- 閉じるボタン
- 設定ボタン

表示例:

```text
共通検索
対象: 管理会社A
[キーワード入力欄] [検索]
[最小化] [閉じる] [設定]
```

### 6.4 最小化

- 最小化時は小さなボタンのみ表示する
- 最小化状態は保存する
- サイトごと、または全体設定として保存できるようにする

### 6.5 ドラッグ移動

初期版では任意実装とする。

実装する場合は、既存入力補助ツールと同様に以下を満たす。

- ドラッグで移動可能
- 移動位置を保存
- 画面外へ出ないように補正
- クリック操作とドラッグ操作を誤判定しない

### 6.6 キーボードショートカット

任意実装とする。

候補:

- `Ctrl + /`: 検索欄へフォーカス
- `Esc`: パネルを閉じる、または検索欄のフォーカス解除
- `Enter`: 検索実行

ショートカットはサイト側ショートカットと競合する可能性があるため、設定でON/OFFできるようにする。

## 7. 検索方式

検索方式は大きく2種類に対応する。

### 7.1 URL直接検索

検索結果URLにキーワードを含められるサイトでは、URLを組み立てて直接遷移する。

例:

```js
searchUrlBuilder(keyword) {
  return `https://example.com/search?keyword=${encodeURIComponent(keyword)}`;
}
```

メリット:

- どのページからでも安定して検索できる
- 検索ページのDOM構造に依存しにくい
- 実行速度が速い

注意点:

- URL仕様が変わると設定更新が必要
- 複雑な検索条件には向かない

### 7.2 検索ページ経由フォーム検索

検索URLにキーワードを直接渡せないサイトでは、一度検索ページへ移動し、検索ページ上の既存フォームに値を入れて検索する。

処理の流れ:

```text
検索パネルでキーワード入力
↓
キーワードを一時保存
↓
検索ページへ遷移
↓
検索ページで一時保存キーワードを取得
↓
検索inputへ値をセット
↓
input / change イベントを発火
↓
検索ボタンをクリック
↓
一時保存キーワードを削除
```

メリット:

- URL検索に対応していないサイトでも利用できる
- サイト本来の検索ロジックを利用できる

注意点:

- 検索ページのDOM構造に依存する
- React / Vue などの制御入力欄では値反映に工夫が必要
- ボタン探索を誤ると意図しない操作になる可能性がある

## 8. サイト別設定

### 8.1 設定形式

サイトごとの差異は `siteConfigs` で管理する。

```js
const siteConfigs = [
  {
    id: 'example-a',
    name: '管理会社A',
    host: 'example-a.com',
    enabled: true,
    mode: 'url',
    searchUrlBuilder(keyword) {
      return `https://example-a.com/search?keyword=${encodeURIComponent(keyword)}`;
    },
  },
  {
    id: 'example-b',
    name: '管理会社B',
    host: 'example-b.com',
    enabled: true,
    mode: 'form',
    searchPageUrl: 'https://example-b.com/properties',
    keywordSelector: 'input[placeholder*="物件名"]',
    buttonSelector: 'button[type="submit"]',
    buttonText: '検索',
  },
];
```

### 8.2 設定項目

| 項目 | 必須 | 内容 |
| --- | --- | --- |
| `id` | 必須 | 設定を識別する一意のID |
| `name` | 必須 | 画面に表示するサイト名 |
| `host` | 必須 | 対象ドメイン |
| `enabled` | 任意 | この設定を有効にするか |
| `mode` | 必須 | `url` または `form` |
| `searchUrlBuilder` | `url` の場合必須 | 検索URL生成関数 |
| `searchPageUrl` | `form` の場合必須 | 検索フォームがあるページ |
| `keywordSelector` | `form` の場合必須 | 検索inputのCSSセレクタ |
| `buttonSelector` | 任意 | 検索ボタンのCSSセレクタ |
| `buttonText` | 任意 | 検索ボタン文言 |
| `notes` | 任意 | 設定メモ |

### 8.3 host判定

標準では以下のように判定する。

```js
location.hostname === config.host || location.hostname.endsWith(`.${config.host}`)
```

サブドメインを含めて同一サイトとして扱えるようにする。

### 8.4 未設定サイト

未設定サイトでは以下の動作とする。

- 検索パネルを表示しない
- 入力欄へ干渉しない
- 検索処理を実行しない
- Tampermonkeyメニューから設定画面を開けるようにするかは後続検討

## 9. 検索実行仕様

### 9.1 共通処理

検索ボタン押下、またはEnterで検索実行する。

処理:

```text
1. 入力値を取得
2. 前後空白を除去
3. 空文字なら実行しない
4. 現在サイトに対応する設定を取得
5. 設定が無効なら実行しない
6. mode に応じて処理を分岐
```

### 9.2 URL直接検索

```text
1. searchUrlBuilder(keyword) でURL生成
2. URLが不正でないか確認
3. location.href で遷移
```

### 9.3 フォーム検索

```text
1. 現在ページが searchPageUrl か確認
2. 違う場合はキーワードを一時保存して searchPageUrl へ遷移
3. 検索ページで一時保存キーワードを取得
4. keywordSelector で検索欄を取得
5. 検索欄へ値をセット
6. input / change イベントを発火
7. 検索ボタンを探索
8. 検索ボタンをクリック
9. 一時保存キーワードを削除
```

## 10. 入力欄への値反映

React / Vue / 独自JSサイトでも反応しやすいよう、既存入力補助ツールで得た知見を取り入れる。

### 10.1 標準反映

```js
input.value = keyword;
input.dispatchEvent(new Event('input', { bubbles: true }));
input.dispatchEvent(new Event('change', { bubbles: true }));
```

### 10.2 setter反映

必要に応じて、HTMLInputElement / HTMLTextAreaElement の native setter を使う。

```js
const setter = Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value')?.set;
setter.call(input, keyword);
```

### 10.3 キーボード風入力

反映されないサイト向けに、後続対応としてキーボード入力風モードを検討する。

初期版では必須としない。

## 11. 検索ボタン探索仕様

検索ボタンは以下の優先順位で探索する。

```text
1. buttonSelector で取得
2. 検索inputが所属するform内のsubmit button
3. buttonText に一致するボタン
4. 「検索」「絞り込み」「表示」「この条件で検索」を含むボタン
```

対象要素:

```text
button
input[type="button"]
input[type="submit"]
```

誤操作防止のため、探索範囲は可能な限り検索inputの近辺または同一form内を優先する。

## 12. 一時保存仕様

フォーム検索でページ遷移を挟む場合、検索キーワードを一時保存する。

### 12.1 保存キー

```js
const PENDING_KEYWORD_KEY = 'floatingPropertySearch.pendingKeyword.v1';
```

### 12.2 保存内容

```js
{
  host: location.hostname,
  configId: config.id,
  keyword,
  createdAt: Date.now()
}
```

### 12.3 読み取り条件

以下を満たす場合のみ使用する。

- hostまたはconfigIdが一致する
- createdAt から10分以内
- keyword が空でない
- 現在ページが対象検索ページである

使用後は必ず削除する。

## 13. 保存データ仕様

保存キーは `floatingPropertySearch.*` に統一する。

| キー | 内容 |
| --- | --- |
| `floatingPropertySearch.panelState.v1` | 最小化状態、位置 |
| `floatingPropertySearch.disabledSites.v1` | サイト別無効化 |
| `floatingPropertySearch.pendingKeyword.v1` | フォーム検索用の一時キーワード |
| `floatingPropertySearch.siteConfigs.v1` | 後続対応の画面編集用サイト設定 |
| `floatingPropertySearch.history.v1` | 後続対応の検索履歴 |

初期版では、サイト設定はコード内定義を基本とする。

## 15. 既存入力補助ツールとの連携仕様

初期版では連携しない。

後続版で以下を検討する。

### 15.1 候補データ連携

検索ランチャーのキーワード入力欄に、既存入力補助ツールの候補を表示できるようにする。

方針:

- 既存ツール側の保存データを直接壊さない
- 読み取り専用から開始する
- 連携ON/OFFを設定で切り替える
- 既存ツールが存在しない場合でも本ツール単体で動作する

### 15.2 使用回数連携

検索ランチャーで検索した語句を、既存入力補助ツールの使用回数へ反映するか検討する。

注意点:

- 入力補助ツール側の使用回数は「実際に検索が行われた時に+1」という仕様
- 本ツールの検索実行成功判定が必要
- 誤記録を避けるため初期版では行わない

### 15.3 共通部品化

将来的に以下を共通化できる。

- Shadow DOM UI基盤
- 入力欄への値セット処理
- React / Vue向けイベント発火
- サイト別ON/OFF
- JSON移行処理
- 検索正規化処理

共通化は、両プロジェクトが安定してから行う。

## 16. 安全設計

### 16.1 誤操作防止

- 未設定サイトでは何もしない
- 空文字検索は実行しない
- URL生成失敗時は遷移しない
- 検索ボタンが見つからない場合はクリックしない
- 複数候補のボタンがある場合は、検索inputと同一form内を優先する
- 実行後は一時保存キーワードを削除する

### 16.2 大量アクセス防止

- 自動連続検索は実装しない
- 1回のユーザー操作につき1回の検索のみ実行する
- 検索結果の自動巡回は行わない

### 16.3 サイト規約配慮

本ツールはユーザーの手動検索操作を補助するものであり、スクレイピングや大量アクセスを目的としない。

## 20. 完了条件

初期版の完了条件は以下。

- 対象サイト内の任意ページで検索パネルが表示される
- 未設定サイトでは何も表示されない
- キーワードを入力して検索できる
- URL検索対応サイトでは検索結果ページへ直接遷移する
- フォーム検索対応サイトでは検索ページへ移動後、自動検索される
- 検索欄への値セット後に input / change イベントが発火する
- 検索ボタンがクリックされる
- 最小化状態が保存される
- サイト別ON/OFFができる
- 無効化後もTampermonkeyメニューから再有効化できる
- 既存入力補助ツールのUI、保存データ、候補ポップアップと競合しない

## 21. 開発時確認項目

サイトごとに以下を確認する。

```text
1. 対象ドメイン
2. 検索結果URLにキーワードが含まれるか
3. 検索ページURL
4. 検索inputのセレクタ
5. 検索ボタンのセレクタ
6. Enter検索が効くか
7. ボタンクリックで検索できるか
8. React / Vue 系の入力欄か
9. 検索ページ以外から検索できるか
10. 既存入力補助ツールと同時に使ってもUIが重ならないか
11. 無効化した時にUIが消えるか
12. Tampermonkeyメニューから再有効化できるか
```
